# PRD: 제안서 자동 검토 시스템

**버전**: 0.1 (초안)
**작성일**: 2026-06-21
**상태**: 검토 중

---

## 1. 배경 및 목적

컨설팅 제안서 제출 전, 담당자가 수작업으로 수행하던 세 가지 검수 작업을 자동화한다.

| 검수 항목 | 현재 방식 | 자동화 목표 |
|---|---|---|
| 최상급 표현 (최초·최대·유일·가장 등) | 육안 검토 | LLM이 플래그, 최종 판단은 사용자 |
| 오타 (한글/영문) | 육안 검토 | LLM이 수정 제안, 채택은 사용자 |
| 회사식별정보 노출 (블라인드 평가 시) | 육안 검토 | 텍스트 매칭 + 이미지 검출 |

**대상 사용자**: 단일 회사 내부 직원 (외부 SaaS 아님)

---

## 2. 릴리즈 단계별 범위

### MVP (v1)
- 회원가입 / 로그인
- PPT/PPTX, DOC/DOCX, PDF 업로드 (정성제안서·정량제안서·발표본 3종)
- LLM 기반 최상급 표현 검출 + 오타 검출
- 검토 결과 화면 (파일·위치 표시)
- 검토 이력 조회

### Phase 2
- 결과 내보내기 (PDF·Excel)
- 블라인드 평가 모드 — 텍스트 기반 회사명·대표자명 검출
- 검토 항목 수락/무시 상태 저장

### Phase 3
- HWP(아래한글) 파일 지원
- 블라인드 평가 — 로고 이미지 검출 (기술 리스크 높음, §7 참조)

### FR-02. 검토 건 생성 및 파일 업로드

**검토 건(Review Job)**: 한 번의 검토 요청 단위. 1~3종류의 파일 묶음.

| ID | 요구사항 |
|---|---|
| FR-02-1 | 검토 건 생성 시 제안서 종류별 파일 업로드: 정성제안서·정량제안서·발표본 |
| FR-02-2 | 지원 형식: .ppt .pptx .doc .docx .pdf |
| FR-02-3 | 정성제안서, 정량제안서는 종류당 여러 파일 허용 (챕터별 분할 대응) |
| FR-02-4 | 발표본은 파일 1개 (추후 다중 파일 허용 여부 결정 필요 — §8 참조) |
| FR-02-5 | 3종류 모두 업로드할 필요 없음. 1종류 이상이면 검토 가능 |
| FR-02-6 | 검토 건 목록에서 각 종류별 파일 수 확인 가능 |
| FR-02-7 | 업로드 후 "검토 시작" 버튼을 누르기 전까지 파일 추가·교체 가능 |
| FR-02-8 | 단일 파일 최대 크기: **50 MB** (결정 필요 — §8 참조) |

---

### FR-03. 검토 실행

**트리거**: 사용자가 "검토 시작" 버튼을 누를 때 실행. 이후 파일 변경 불가.

**처리 흐름**:
```
파일 파싱 (텍스트·위치 추출)
  → 청크 분할 (LLM 컨텍스트 제한 대응)
  → LLM 호출 (최상급 표현 + 오타 동시 분석)
  → 결과 저장
```

| ID | 요구사항 |
|---|---|
| FR-03-1 | 검토는 비동기로 처리. 사용자는 화면을 벗어나도 결과 조회 가능 |
| FR-03-2 | 검토 상태 표시: 대기 중·처리 중·완료·오류 |
| FR-03-3 | LLM은 각 파일의 텍스트를 받아 최상급 표현과 오타를 동시에 분석 |
| FR-03-4 | 검출 결과에는 **파일명 + 페이지/슬라이드 번호**를 포함 |
| FR-03-5 | 처리 오류 시 오류 메시지와 함께 재시도 버튼 제공 |
| FR-03-6 | 처리 타임아웃: 검토 건 전체 기준 **30분** (결정 필요 — §8 참조) |

**LLM 분석 지침 (프롬프트 설계 방향)**:
- 최상급 표현: "최초·최대·최고·유일·가장·독보적" 등을 검출하되, 사실 여부를 판단하지 않고 "검토 필요" 플래그만 부여
- 오타: 한글·영문 맞춤법 오류를 검출하고 수정 제안 1~3개를 함께 반환
- 오탐 최소화를 위해 고유명사·약어·브랜드명은 오타로 처리하지 않도록 지침 포함

### FR-04. 검토 결과 화면

**그룹핑 구조**:
```
검토 건
 ├── 정성제안서
 │    ├── 01_서론.pptx   (최상급 표현 N건, 오타 N건)
 │    └── 02_본론.pptx   (...)
 ├── 정량제안서
 │    └── 정량.pdf       (...)
 └── 발표본
      └── 발표.pptx      (...)
```

| ID | 요구사항 |
|---|---|
| FR-04-1 | 결과는 "검토 건 → 제안서 종류 → 개별 파일" 순으로 그룹핑 |
| FR-04-2 | 최상급 표현 / 오타 섹션을 구분하여 표시 |
| FR-04-3 | 각 항목: 검출 텍스트, 파일명, 페이지/슬라이드 번호 표시 |
| FR-04-4 | 오타 항목: 수정 제안 텍스트도 함께 표시 |
| FR-04-5 | MVP에서는 항목별 수락/무시 처리 없음 (읽기 전용) → Phase 2에서 추가 |
| FR-04-6 | 종류별·파일별 검출 건수 요약을 상단에 표시 |

## 4. 비기능 요구사항

### 4-1. 보안

| 항목 | 요구사항 |
|---|---|
| 파일 접근 | 업로드 파일은 업로드한 사용자만 접근 가능 (URL 직접 접근 차단) |
| 전송 암호화 | 모든 통신 HTTPS |
| 파일 보관 | 검토 완료 후 **90일** 보관 후 자동 삭제 (결정 필요 — §8 참조) |
| 패스워드 | bcrypt 등 단방향 해시 저장 |

> 제안서는 기업 민감정보를 포함할 수 있으므로, 파일 저장 경로는 외부 추측이 불가능한 방식(UUID 기반 경로)으로 관리한다.

### 4-2. 성능

| 항목 | 목표 |
|---|---|
| 파일 업로드 응답 | 50 MB 이하 파일 기준 5초 이내 |
| 검토 처리 시간 | 파일 총합 100페이지 기준 **10분 이내** 완료 목표 |
| 동시 검토 건 | MVP 기준 **5건 동시 처리** (큐 기반 비동기 처리로 확장 가능) |

### 4-3. 가용성·운영

- 검토 실패 시 자동 재시도 1회 후 오류 상태로 전환, 사용자에게 알림
- 로그: 검토 요청·완료·오류 이벤트 기록 (장애 원인 추적용)

## 7. 기술 리스크

### R-01. HWP 파일 파싱 (Phase 3, 위험도: 높음)
- HWP는 독점 바이너리 포맷으로 오픈소스 파서가 제한적
- **검토 필요 라이브러리**: `python-hwp`(비공식), `hwp5` (LibreOffice 경유 변환), 한컴 공식 SDK (라이선스 필요)
- **권장 접근**: LibreOffice를 서버에 설치하여 HWP → DOCX 변환 후 처리. 단, 수식·특수 레이아웃 손실 가능
- **결정**: Phase 3 착수 전 파싱 가능 범위를 PoC로 검증 필요

### R-02. 로고 이미지 검출 (Phase 3, 위험도: 높음)
- 이미지 유사도 매칭(템플릿 매칭 또는 임베딩 벡터 비교) 필요
- 로고가 워터마크·배경 처리된 경우 검출율 크게 저하
- PPT/PDF 내 이미지 추출 자체도 라이브러리별 품질 차이 존재
- **권장 접근**: MVP에서 완전히 제외, Phase 3에서 별도 기술 검증 후 포함 여부 결정
- **사용자 안내 문구 필수**: "해상도·압축 수준에 따라 로고가 검출되지 않을 수 있습니다"

### R-03. LLM 오탐 (MVP, 위험도: 중간)
- 최상급 표현: 고유명사나 인용문 내 표현을 오탐할 수 있음
- 오타: 전문 용어·약어·브랜드명을 오탐할 수 있음
- **완화 방안**: 프롬프트에 예외 처리 지침 포함, 오탐 사례 누적 후 프롬프트 개선 반복
- **설계 원칙**: LLM 결과는 항상 "검토 필요" 수준으로 제시. 자동 수정 없음

### R-04. 대용량 파일 처리 (MVP, 위험도: 중간)
- 100페이지 이상 PDF나 고해상도 이미지 포함 PPT는 파싱·LLM 호출 시간이 급격히 증가
- **완화 방안**: 비동기 큐 처리, 페이지 단위 청크 병렬 LLM 호출, 타임아웃 후 부분 결과 반환 고려

### R-05. 파일 파싱 정확도 (MVP, 위험도: 중간)
- PPT: 텍스트 박스 순서, 마스터 슬라이드 텍스트 추출 정확도
- PDF: 스캔 PDF(이미지 기반)는 OCR 없이 텍스트 추출 불가
- **완화 방안**: 스캔 PDF는 오류 메시지로 안내 ("텍스트 레이어가 없어 분석할 수 없습니다")

## 9. 제외 범위 (명시적)

- 제안서 파일 자동 수정 (검출 및 제안만 제공, 수정은 사용자 직접 수행)
- 팀·조직 단위 권한 관리 (개인 계정 단위만)
- 실시간 협업·코멘트 기능
- 모바일 앱
- 외부 시스템 연동 (이메일 알림, Slack 등)
